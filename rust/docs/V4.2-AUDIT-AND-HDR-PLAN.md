## BLOCKING STRATEGIC FINDING — do NOT merge the fork wholesale

**The fork is still v4.1-based.** `hdr/main`'s `EbSvtAv1.h` reads `MINOR 1`;
`git merge-base --is-ancestor v4.2.0-rc hdr/main` = **NO**; merge-base(`hdr/main`,`v4.2.0-rc`)
= `c04f9515` = **mainline v4.1.0**. Therefore the directive's "retarget the C baseline to the
fork" **must not** be done as a merge of `hdr/main` — that would regress the in-tree C baseline
from 4.2.0-rc back to a 4.1 base and silently undo the whole v4.2 cycle the port is now
validated against.

**Correct approach — rebase the fork delta onto v4.2.0-rc:**

Use the **pure fork delta** `mainline v4.1.0 (c04f9515) .. hdr-v4.1.0` (5109 lines, 41 files
under `Source/`). Do **not** use `v4.1.0..hdr/main` — that range also contains the fork's
cherry-picks of mainline 4.2-cycle work, which would double-apply against our 4.2 tree.

**Measured cost (`git apply --3way --check`, tree unmodified):**

```
28 / 41 files apply CLEANLY
11 files conflict:
  Source/Lib/Codec/rc_aq.c               <- PQ curve-3 (the flagship)
  Source/Lib/Codec/product_coding_loop.c <- psy RD (ac-bias/tx_bias/complex_hvs)
  Source/Lib/Codec/full_loop.c           <- noise-norm AC boost
  Source/Lib/Codec/enc_mode_config.c     <- Tune-5 + per-preset gating (v4.2 rewrote +8688 lines here)
  Source/Lib/Codec/{definitions.h,pcs.h} <- struct/macro plumbing
  Source/Lib/Globals/{enc_settings.c,enc_handle.c}, Source/API/EbSvtAv1Enc.h <- config fields
                                            (v4.2 added its own new fields in the same regions)
  Source/App/{app_config.c,app_main.c}   <- CLI
```
All 11 are tractable: mostly "both sides added new config fields adjacent to each other".

## Why the C side matters for a Rust port

The port proves identity by differentially testing against the linked in-tree `libSvtAv1Enc.a`.
To port HDR features **under the same bit-identity mandate**, we need a C library that HAS
those features — i.e. a **"v4.2.0-rc + fork delta" hybrid C baseline** built in this worktree.
Porting HDR features to Rust without a C oracle would violate the mandate.

## Sequenced plan (Phase B, gated on Phase A completing)

1. Rebase the pure fork delta onto the v4.2.0-rc tree here; resolve the 11 conflicts
   (config-plumbing first, then the 4 algorithm files). Build → hybrid `libSvtAv1Enc.a`.
2. Verify the hybrid: fork features OFF must reproduce v4.2.0-rc byte-for-byte on the
   existing identity matrix (proves the rebase is behavior-neutral when disabled).
3. Extend `svtav1-cref` to expose the fork entry points; add `c_parity_*` tests per feature.
4. Port to Rust in this order (self-contained → invasive):
   **PQ variance-boost curve 3** (`summaries/hdr_curve3_rc_aq_function.c`, ~140 lines,
   inputs = per-SB variance octiles + mean luma + base qindex → deltaq; auto-selects at
   `transfer_characteristics == 16`) → `--noise*` in-encoder grain synthesis
   (`noise_generation.c`, NEW 324 lines) → `--cdef-scaling` → Tune-5 policy →
   `tx_bias` / `complex_hvs`.
   **Prerequisite for curve 3:** a C-exact `rc_aq.c` port must exist first — today the port has
   no C-exact AQ at all (only the dormant zenrav1e `perceptual.rs`). Curve 3 is an additive
   branch on top of C's variance boost (curves 0/1/2, already in mainline v4.1+).
5. Keep fork features **opt-in** behind config flags; mainline-default output must stay
   byte-identical to mainline v4.2.0-rc.
